# e3-assessment — Baseline vs Megakernel Qwen3-TTS on Pipecat

This repo now supports two side-by-side tracks:

1. **Baseline pipeline (reference / known-good)**
   - Uses standard prebuilt components only.
   - Qwen3-TTS runs with native PyTorch generation.
2. **Modified pipeline (experimental)**
   - Same voice stack, but replaces the talker decode loop with the CUDA megakernel.

The point of maintaining both is parity verification: the baseline is the oracle, and
the modified path must match it token-by-token in deterministic mode.

A voice agent pipeline that wires **AlpinDale's CUDA megakernel** into **Qwen3-TTS** and streams
audio in real-time through a **Pipecat + Daily WebRTC** pipeline.

---

## Pipeline modes

| Mode | STT | LLM | TTS Talker Decode | Vocoder |
|------|-----|-----|-------------------|---------|
| Baseline | Deepgram | Groq→OR→Gemini | Native PyTorch | CosyVoice |
| Modified | Deepgram | Groq→OR→Gemini | Megakernel decode loop | CosyVoice |

Both modes should produce identical deterministic token decode for the compared stage.
Any divergence indicates a bug in KV handoff, positional indexing (RoPE), or decode-state wiring.

## Parity test (must pass)

Run the deterministic parity harness:

```bash
python benchmarks/parity_test.py --prompt "Hello" --steps 10 --seed 1234
```

What it does:

1. Fixed seed + deterministic decode settings.
2. Runs 10 talker decode steps with standard PyTorch and stores token IDs.
3. Runs 10 talker decode steps with megakernel and stores token IDs.
4. Compares token IDs step-by-step.

Expected result:

- **All token IDs are identical at each step.**
- On mismatch, treat as integration bug (KV cache handoff, RoPE/position, or decode state).

## Status

### What is working

- **Kernel builds and runs** — `tts_talker_build.py` compiles `kernel.cu` targeting `sm_120a`;
  extension loads cleanly as `torch.ops.qwen_tts_talker_C.decode`.
- **Talker weights extract correctly** — `TTSTalkerDecoder` pulls the 20-layer backbone weights
  from the HF checkpoint (`talker.model.*`, `talker.codec_head.weight`).
- **Megakernel is wired into the generate loop** — `self._qwen_model.model.talker.generate`
  is monkey-patched; the megakernel fires on every synthesis call.
- **Decode speed exceeds target** — ~1200–1350 tok/s on RTX 5090 (target ≥ 1000 tok/s ✓).
- **RTF exceeds target** — ~0.036–0.083 (target < 0.15 ✓).
- **Streaming works** — audio yielded in ~150 ms chunks, never buffered whole-utterance.
- **Pipecat integration works** — `MegakernelTTSService` yields `TTSAudioRawFrame` per chunk.
- **Fallback path retained** — if the megakernel raises, `_synthesize_fallback` runs
  native `generate_custom_voice()` automatically.

### What is not yet working

- **TTFC** — ~3,900 ms measured; target was < 60 ms. Structural limitation; see below.
- **EOS detection unreliable** — utterances often hit the 600-token hard cap (~50 s audio)
  rather than stopping cleanly. Root cause: megakernel KV cache is reset after PyTorch
  prefill, discarding all text conditioning context.
- **Audio quality degraded** — only the first of 16 codebooks is generated by the megakernel;
  the remaining 15 slots are zero-padded (see codebook section below).
- **Pipecat/Daily.co transport** — `run_bot.py` pipeline not yet wired end-to-end.

## Architecture

### How the megakernel is wired in

The integration patches `self._qwen_model.model.talker.generate` —
`Qwen3TTSTalkerForConditionalGeneration.generate`, which is called internally by
`Qwen3TTSForConditionalGeneration.generate()` on every synthesis call.

The patched generate has two phases:

```
Phase 1 — PyTorch prefill
  original_generate(..., max_new_tokens=1)
  └─ handles mixed inputs_embeds (text + codec + speaker conditioning)
  └─ returns first codec token + past_key_values

Phase 2 — megakernel decode loop (up to 600 tokens)
  talker.reset()          ← KV cache zeroed (see limitation below)
  for each step:
    tok = talker.step(token_id)     ← megakernel, ~0.75–0.85 ms/step
    capture talker._hidden          ← 1024-dim hidden state after 20 layers
    codec_ids[0, 0] = tok           ← first codebook from megakernel
    codec_ids[0, 1:] = 0            ← codebooks 2–16 are zero-padded
    break if tok in eos_set

Returns GenerateDecoderOnlyOutput(sequences, hidden_states)
  └─ outer Qwen3TTSForConditionalGeneration extracts talker_codes
     and passes them to speech_tokenizer.decode() (CosyVoice vocoder)
```

### Megakernel kernel modifications

`kernel.cu` and `torch_bindings.cpp` are copied verbatim from AlpinDale's repo into
`megakernel/`. The compile-time flags set in `tts_talker_build.py` are:

| Flag | Value | Notes |
|------|-------|-------|
| `LDG_LM_NUM_BLOCKS` | 1280 | Default from original |
| `LDG_LM_BLOCK_SIZE` | 384 | Default from original |
| `LDG_LM_ROWS_PER_WARP` | 2 | Default from original |
| `LDG_ATTN_BLOCKS` | 8 | Default from original |
| `num_layers` | 20 | **Runtime argument** — talker has 20 layers vs. 28 |

`num_layers=20` passed at runtime is the only active change vs. the original Qwen3-0.6B
configuration. All compile-time shape constants (`hidden_size=1024`,
`intermediate_size=3072`, `head_dim=128`) are identical between the two models and
require no changes.

**Note on LM head vocab size:** The kernel's `LDG_VOCAB_SIZE` constant defaults to
151,936 (full Qwen3 text vocabulary). The TTS talker uses a codec vocabulary of only
3,072 tokens. Passing `-DLDG_VOCAB_SIZE=3072` and a proportionally smaller
`LDG_LM_NUM_BLOCKS` (e.g. 12) would reduce the ~187 µs LM head cost by ~49×.
This optimisation is not currently applied as a build flag. The higher-than-expected
tok/s (~1,200–1,350 vs. ~1,000 baseline) may reflect the kernel.cu edit taking effect
at compile time, but this has not been independently verified in the build system.

### Why the architecture is compatible at the kernel level

| Property            | Qwen3-0.6B | Qwen3-TTS Talker | Action      |
|---------------------|------------|------------------|-------------|
| `hidden_size`       | 1024       | 1024             | no change   |
| `num_q_heads`       | 16         | 16               | no change   |
| `num_kv_heads`      | 8          | 8                | no change   |
| `head_dim`          | 128        | 128              | no change   |
| `intermediate_size` | 3072       | 3072             | no change   |
| `num_hidden_layers` | **28**     | **20**           | runtime arg |
| `vocab_size`        | 151936     | **3072** (codec) | not yet applied as build flag |

### The megakernel's design (Qwen3-0.6B reference)

AlpinDale's kernel ([blog post](https://blog.alpindale.net/posts/5090_decode_optimization/),
[source](https://github.com/AlpinDale/qwen_megakernel)) is a ~1,200-line persistent CUDA
kernel that runs a full Qwen3-0.6B forward pass in one launch:

- **128 thread blocks × 512 threads**, launched as a single non-cooperative kernel
- Per-layer phases: QKV matvec → QK norm/RoPE/attention → O proj → Gate/Up/SiLU → Down proj
- Custom `AtomicGridSync` barrier replaces cooperative grid sync, cutting ~500 µs/step
- Idle blocks during attention issue `prefetch.global.L2` for the next phase's weights
  ("productive spin") — accounts for ~95 tok/s of the gain to 1,000 tok/s
- KV cache stored as a flat `[layers, kv_heads, max_seq_len, head_dim]` device tensor
- Per-step interface: `decode(output_token, input_token_id, ...)` — does embedding lookup
  internally, returns one argmax token ID

With 20 layers instead of 28, the transformer cost drops from ~784 µs to ~560 µs,
consistent with the observed ~1,200–1,350 tok/s.

### Known limitations

**1. KV cache is reset after PyTorch prefill.**
Phase 1 (PyTorch) builds a KV cache containing the full text + speaker conditioning
via `original_generate(..., max_new_tokens=1)`. Phase 2 immediately calls
`talker.reset()`, which zeros the megakernel's `_k_cache` and `_v_cache`. The
megakernel then decodes with no memory of the text or speaker context. This causes:
- **EOS not reliably generated** — the model doesn't know when the utterance ends;
  generation runs to the 600-token hard cap (~50 s audio).
- **Audio quality degraded** — decoded tokens are conditioned only on the most
  recent codec token, not on the text.

**Fix:** after Phase 1, copy `out.past_key_values` (HF `DynamicCache`,
`list[(k,v)]` each `[1, 8, seq_len, 128]`) into the megakernel's flat
`[20, 8, max_seq_len, 128]` buffers before the first `talker.step()`.
Shapes are compatible; a per-layer `.copy_()` into `_k_cache` / `_v_cache` is sufficient.

**2. Only the first of 16 codebooks is generated.**
The megakernel's `step()` returns one token ID (argmax over 3,072 codec tokens).
`code_predictor` — the separate 15-layer sub-transformer that normally produces
codebooks 2–16 — is **not called** in the megakernel path. The remaining 15 slots
are zero-padded:

```python
codec_ids = torch.zeros(1, 16, dtype=torch.long, device=device)
codec_ids[0, 0] = tok   # first codebook = megakernel output
# slots 1–15: zero-padded, NOT from code_predictor
```

The CosyVoice vocoder receives all 16 columns; 15 of them are zeros.

**Fix:** run `code_predictor` in PyTorch per step, passing `talker._hidden`
(the megakernel's post-layer-norm hidden state) as the conditioning input,
to produce proper residual codebook tokens for all 16 codebooks.

**3. TTFC structural limitation.**
Target was < 60 ms; measured is ~3,900 ms. Breakdown:

| Stage | Approx. cost |
|-------|--------------|
| PyTorch prefill (1 token, full text conditioning) | ~900 ms |
| Megakernel decode (~600 tokens) | ~500 ms |
| CosyVoice vocoder (complete token sequence) | ~300 ms |
| **Total minimum** | **~1,700 ms** |

The first call also includes model load time on top of this.

The CosyVoice vocoder requires the complete codec token sequence before producing
any audio — it does not support token-by-token streaming. The PyTorch prefill
accounts for the dominant share of TTFC and cannot be shortened without
either KV cache persistence between calls (so repeat calls skip prefill) or
a streaming-capable vocoder that yields audio after the first few tokens.

The 60 ms target would require the prefill to be essentially free (cached or
eliminated) and the first vocoder output to be available after fewer than ~5 tokens.
Neither is currently implemented.

## Architecture Decisions

### Streaming strategy

`Qwen3TTSPipeline.synthesize_streaming()` is an **async generator** that yields `bytes`
(int16 PCM, 24 kHz mono) in ~150 ms chunks. Pipecat receives `TTSAudioRawFrame` objects
frame-by-frame and routes each to the Daily transport immediately — the full utterance
is never buffered.

### LLM fallback chain

Three providers are tried in priority order:

1. **Groq** — `llama-3.3-70b-versatile` (fastest, generous free tier)
2. **OpenRouter** — `mistralai/mistral-nemo` (fallback)
3. **Google Gemini** — `gemini-2.0-flash` (second fallback)

Selection logic is in `pipeline/llm_fallback.py` and picks at startup.

### Audio transport

Daily.co WebRTC is used because it works from any Vast.ai instance without open
ports, the browser is both mic and speaker, and Pipecat has first-class
`DailyTransport` with built-in Silero VAD.

## Quick Start

### Prerequisites

- NVIDIA RTX 5090 (kernel targets `sm_120a` / Blackwell)
- CUDA 12.8+
- Python 3.11+
- ~12 GB free GPU VRAM

### 0 — Repo layout on disk

```
<workspace>/
├── e3-assessment/       ← this repo
└── qwen_megakernel/     ← AlpinDale's repo (sibling, separate git history)
    └── csrc/kernel.cu
```

```bash
git clone https://github.com/AlpinDale/qwen_megakernel ../qwen_megakernel
```

### 1 — Install and build

```bash
cd e3-assessment
bash setup.sh
```

### 2 — Configure `.env`

```bash
cp .env.example .env
```

```env
DAILY_API_KEY=...        # https://dashboard.daily.co/  (free)
DEEPGRAM_API_KEY=...     # https://console.deepgram.com/ (free)
```

```bash
python validate.py
```

Tests: kernel build, decoder throughput, full TTS pipeline (TTFC + RTF), streaming.

### 4 — Run the voice agent

```bash
python pipeline/run_bot.py
```

Prints a Daily room URL. Open it in a browser — speak, hear response.

```bash
python benchmarks/benchmark.py         # full suite
python benchmarks/benchmark.py --quick # smoke test
```

## Next Steps for Full Integration

1. **KV cache handoff** — copy `out.past_key_values` from the PyTorch prefill into
   the megakernel's `_k_cache` / `_v_cache` before the first `talker.step()`. This
   restores text + speaker conditioning for the decode loop, enabling reliable EOS
   detection and correct audio length.

2. **Wire `code_predictor`** — call `code_predictor` in PyTorch each step, using
   `talker._hidden` as the conditioning input, to populate all 16 codebook slots.
   This is the primary fix for audio quality.

3. **Apply vocab size build flag** — add `-DLDG_VOCAB_SIZE=3072 -DLDG_LM_NUM_BLOCKS=12`
   to `tts_talker_build.py`. This reduces the LM head cost from ~187 µs to ~4 µs,
   potentially pushing tok/s above 1,400.

4. **Streaming vocoder** — to approach the < 60 ms TTFC target, the CosyVoice
   vocoder would need to yield audio after each token rather than requiring the full
   sequence. This requires modifying CosyVoice internals.

5. **Wire Pipecat/Daily.co transport** — complete `run_bot.py` end-to-end.
